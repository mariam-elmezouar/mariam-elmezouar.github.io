Labo 7: Conception d un reseau d entreprise

GEF330 hiver 2021

Objectifs

Le but de ce labo de conception au niveau de l’entreprise (EDX - Entreprise Design Exercise) est de vous demander :

  • d’effectuer une conception non-dirigée sur un problème compliqué et réaliste.
  • de faire la conception, la construction et le testage d’un réseau de niveau d’entreprise qui rencontre les exigences données, en respectant des limitations de ressources et d’autres contraintes.

Lorsque vous parcourez les exigences du lab, ne paniquez pas ! Oui, on vous en demande beaucoup, mais ces serveurs ont déjà été mis en place pour vous, vous n’avez qu’à vous occuper de leur mise en réseau.

Situation

Votre équipe doit fournir du support d’ingénierie de réseau informatique à une force opérationnelle déployée. La force opérationnelle de coalition (FOC), où chaque équipe représente une force opérationnelle nationale, sera déployé dans le cours d’une opération de restauration de la paix sanctionnée par les NU dans un pays au bord de la guerre civile. Votre force opérationnelle nationale (FON) fera partie d’une coalition internationale. La langue de communication de la FOC sera l’anglais.

Pour supporter les opérations de la FOC, chaque FON connectera son réseau informatique déployé pour créer une infrastructure de coalition pour supporter l’administration et la planification des forces déployées. Vous devez faire la conception et la construction d’un réseau pour supporter votre FON et connecté celui-ci aux autres nations dans un réseau au niveau de la coalition.

Bien que chaque FON est déployée pour collaborer et participer à des opérations interarmées, votre nation n’a pas confiance complète en ses partenaires des autres nations. Vous devez collaborer avec les autres FON d’une part, tout en protégeant votre réseau et vos ressources de ces partenaires.

Spécifications d’exigences réseau

La FON doit fournir les facilités/services suivants. Les exigences de sécurités pour ces facilités/services sont considérées essentielles (elles doivent être atteintes).

Exigences générales

  • Chaque FON se connectera au commutateur principal de la coalition (Coalition Main Switch) dans le sous-réseau 10.12.0.0/16
  • Chaque FON recevra 2 adresses IP sur ce sous-réseau :
    • Votre passerelle principale (Main Gateway) au 10.12.0.X (où ‘X’ est 100+x)
    • Votre serveur web (Web Server) au 10.12.0.Y (où ‘Y’ est 200+x)
  • Votre réseau de FON utilisera une plage d’adresses privée (c.a.d. qu’aucune route à ces adresses n’existe des autres partenaires de la FOC).
    • Tout vos sous réseaux de FON configurés doivent être comprises dans la plage d’adresses 192.168.0.0/24
    • Vous pouvez sous-diviser cette plage d’adresse comme vous le voulez pour répondre aux exigences.

Service Web

Vous devez fournir un serveur Web (Web Server) qui représente un point d’accès pour l’information de la FOC. Le service peut être utiliser par les utilisateurs de votre FON et par ceux des autres membres de la FOC. Le Web Server sert d’interface pour la présentation de l’information que votre FON maintien sur un Operations File Server.

  • Vous devez supporter un hôte pour un Web Server offrant un service HTTP sur le port TCP 80.
    • Les utilisateurs des autres FON de la FOC doivent avoir accès au service HTTP en se connectant directement à votre Web Server.
    • Les utilisateurs de votre FON doivent avoir accès au service HTTP en se connectant de par le Web Proxy Server.
  • Le Web Server doit pouvoir se connecter au Operations File Server que vous maintenez sur le réseau au port TCP 22.
  • Le Web Server ne peux pas initier de connections à tout poste de travail ou serveur, sauf le Operations File Server ou le Web Proxy Server.

Services mandataires HTTP

Votre réseau FON doit fournir un service mandataire HTTP (Web Proxy Server) qui permettra aux utilisateurs de votre FON d’avoir accès au service Web de la FOC, comprenant le Web Server de votre FON et ceux des autres FON.

  • Vous devez supporter un hôte pour un Web Proxy Server offrant un service mandataire HTTP au port TCP 3128.
  • Aucun hôte des autres FON de la FOC ne doit pouvoir initier une connexion au Web Proxy Server.
  • Le Web Proxy Server ne doit pas pouvoir initier une connexion à un User Workstation ou serveur de votre réseau FON autre que le Web Server de la FON.
  • Les User Workstations de votre FON peuvent se connecter au port TCP 3128 du Web Proxy Server pour demander une page Web qui existe dans le réseau de la FOC. Plus précisément, ceci veut dire que toutes les requêtes HTTP faites pare les utilisateurs de toute FON pour obtenir une page (GET) doivent être dirigée au Web Proxy Server au port TCP 3128.
  • Lorsqu’une page Web est demandée, le Proxy Web Server initiera une connexion TCP au port 80 du Web Server de la FON appropriée et demandera la page Web de la part du User Workstation ayant fait la requête initiale.

Service de fichiers opérationnel

Le dépôt principal d’information que vous partagez avec la FOC est stocké dans le serveur de fichier opérationnel (Operations File Server). Les utilisateurs de votre FON maintiennent l’information du Operations File Server à jour. Le Web Server de votre FON accède également le Operations File Server pour disséminer l’information aux utilisateurs de la FOC.

  • Vous devez supporter un hôte pour un Operations File Server offrant un service SSH au port TCP 22.
    • Le service SSH doit être disponible pour tout User Workstation de votre FON.
  • Le Operations File Server ne doit pas pouvoir initier de connexion avec un serveur ou avec les Admin Workstations.
  • Le Web Server doit pouvoir se connecter au Operations File Server au port TCP 22.
  • Les utilisateurs des autres FON ne doivent pas pouvoir initier de connexion au Operations File Server. Vous noterez par contre, que la fuite de l’information qui est située sur le Operations File Server ne peut pas endommager vos intérêts nationaux.

Service de fichiers restreint

Le dépôt principal d’information nationale protégée est le serveur de fichiers restreint (Restricted File Server). Les utilisateurs de votre FON utilisent l’information du Restricted File Server et tiennent celle-ci à jour.

  • Vous devez supporter un hôte pour un Restricted File Server offrant un service SSH au port TCP 22.
    • Le service SSH doit être disponible pour tout User Workstation de votre FON.
  • Le Restricted File Server ne doit pas pouvoir initier de connexions à tout autre serveur or poste de travail.
  • Les User Workstations de votre FON ne doivent pas pouvoir initier de connexions avec le Restricted File Server à un port autre que le TCP 22.
  • Les utilisateurs des autres FON ne doivent pas avoir accès à ce Restricted File Server; une fuite de l’information protégée y résidant entraînerait des dommages à vos intérêts nationaux.

Postes de travail utilisateurs

Les utilisateurs de votre FON, que ce soit des officiers d’état-major, des analystes, des opérateurs, etc., utiliseront des poste de travail pour utilisateurs (User Workstations).

  • Les User Workstations seront des hôtes Linux fournissant une suite d’outils de bureau.
  • Aucun hôte des autres FON ne peut initier de connexion aux User Workstations.
  • Les User Workstations ne doivent pas pouvoir initier de connexions à un hôte sur une autre FON.
  • Les User Workstations peuvent demander des pages Web situées les serveurs HTTP de la FOC en utilisant le Web Proxy Server. Plus précisément, toutes demande de page Web doivent être acheminée au Web Proxy Server au port TCP 3128; le Web Proxy Server fera la demande de page au Web Server approprié de la part du User Workstation.
  • Les User Workstations peuvent se connecter au Operations File Server au port TCP 22.
  • Les User Workstation peuvent se connecter au Restricted File Server au port TCP 22.
  • Votre réseau doit supporter jusqu’à 80 User Workstations.

Postes de travail administratifs

Les administrateurs réseau de votre FON seront équipés de postes de travail administratifs (Admin Workstations). Ils utiliseront ceux-ci pour administrer l’infrastructure réseau de votre FON.

  • Les Admin Workstations seront des hôtes Linux.
  • Aucun hôte des autres FON ne doit pouvoir initier de connexion aux Admin Workstations.
  • Les Admin Workstations ne doivent pas pouvoir initier de connexion à aucun hôte des autres FON.
  • Les Admin Workstations doivent pouvoir initier une connexion SSH avec les coupe-feux (Firewalls) de votre FON pour les gérer.
  • Les Admin Workstations doivent pouvoir initier une connexion SSH avec tous les postes de travails et serveurs de votre FON sur le port TCP 22, incluant:
    • Web Server,
    • Web Proxy Server,
    • Operations File Server,
    • Restricted File Server, et
    • Firewalls.
  • Les hôtes suivant ne doivent pas se connecter aux Admin Workstations :
    • n’importe quel hôte d’une autre FON,
    • les serveurs de votre FON, ou le User Workstations
  • Votre réseau doit pouvoir supporter jusqu’à 5 Admin Workstations.

Les Admin Workstations doivent aussi avoir un adaptateur External. Cet adaptateur doit se voir assigner l’adresse IP 10.12.0.z/16 (où ‘Z’ = 150 + x); cet adaptateur sera utilisé pour valider vos configurations réseau.

Afin de faciliter le test pratique de votre réseau ainsi que pour vous faciliter un peu la vie, vous devez configurer votre Admin Workstation pour pouvoir vous connecter à toutes vos VMs via une clef SSH. Cela permet de vous connecter sans avoir à entrer systématiquement un mot de passe. Executez les commanders suivantes sur votre Admin Workstation :

  • ssh-keygen
    • Acceptez le dossier par défaut et utiliser un mot de passe vide.
    • Ne Faites cette étape qu’une fois.
  • ssh-copy-id -i ~/.ssh/id_rsa.pub gef330@<IP de l'hôte>
    • Testez pour être certain que ça marche.
      • ssh gef330@<IP de l'hôte>
    • Répétez cette étape pour chaque machine que vous devez accèder via SSH.

NOTEZ BIEN qu’il est absolument crucial que vos Admin Workstations soient bien configurées. Vous ne recevrez aucune note si le serveur central ne peut pas se connecter aux Admin Workstations avec SSH au port TCP 22; vous ne recevrez aucune note pour les portions de votre réseau où les Admin Workstations ne peuvent pas se connecter aux autres postes de travail et serveur avec SSH au port TCP 22.

Tâches

Vous devez faire la conception et la construction du réseau requis dans le CNSl (S4112) ou bien l'environnement virtuel équivalent.

Vous avez accès à:

  • L’infrastructure principale du labo (c.à.d. le réseau 10.12.0.0/16 le concentrateur principal, etc.)
  • Vous devez bâtir votre réseau FON dans l’environnement VMWare; notez que vous utiliserez un compte différent pour l’EDX :
    • Les utilisateurs sur le campus utiliseront un nouveau compte d'utilisateur pour cet exercice de conception d'entreprise (EDX).
      • Username: gef330_edx
      • Password: même que pour le compte gef330
    • Les utilisateurs à distance utiliseront les mêmes informations d'identification que leur environnement normal.
  • 7 commutateurs virtuels, étiquetés : External, Alfa, Bravo, Charlie, Delta, Echo et Foxtrot
    • Chacune de vos hôtes auront adaptateurs étiquetés avec les noms de ces commutateurs virtuels. Pour connecter un adaptateur au commutateur, vous devez l’allumer avec :
      • sudo ip link set [interface name] up
    • Les instructeurs s’assureront que vous n’avez allumé que le nombre permis d’adaptateurs comme décrit plus bas.
  • 9 machines virtuelles sur votre hôte VMWare :
    • 3 Firewalls - Vous devez utilisez tout les trois :
      • External Firewall - Coupe-feu externe ayant un adaptateur externe (étiqueté External) et jusqu’à deux autre adaptateurs virtuels, sur des commutateurs distincts.
      • Firewall 2 - Coupe-feu ayant jusqu’à trois adaptateurs virtuels, sur des commutateurs distincts.
      • Firewall 3 - Coupe-feu ayant jusqu’à trois adaptateurs virtuels, sur des commutateurs distincts.
    • Web, pour le Web Server - un adaptateur sur un commutateur virtuel.
    • Proxy, pour le Proxy Server - un adaptateur sur un commutateur virtuel.
    • SSH Restricted pour le Restricted File Server - un adaptateur sur un commutateur virtuel.
    • SSH Operations pour le Operations File Server - un adaptateur sur un commutateur virtuel.
    • User pour le User Workstation - un adaptateur sur un commutateur virtuel.
    • Admin pour le Admin Workstation - un adaptateur sur un commutateur virtuel.

SSH

L'accès à distance SSH doit être activé sur chaque machine virtuelle de votre réseau, le trafic SSH étant contrôlé par vos pare-feu comme spécifié ci-dessus. En plus de permettre l'administration depuis les VM Admin, il est nécessaire pour le test pratique. Vous pouvez démarrer, arrêter, redémarrer et vérifier l'état du SSH daemon sur votre machine virtuelle avec les commandes suivantes:

  • sudo systemctl start sshd
  • sudo systemctl stop sshd
  • sudo systemctl restart sshd
  • sudo systemctl status sshd

Par défaut, SSH est désactivé au démarrage. Vous pouvez l'activer de manière permanente au démarrage avec la commande suivante:

  • sudo systemctl enable sshd

Une fois que cela est fait, SSH sera activé au démarrage. Il n'est pas nécessaire de l'avoir dans votre script.

Pour faciliter les tests de votre réseau, vous allez configurer vos postes de travail d' administrateur avec des clés d'accès SSH à toutes vos machines virtuelles. Cela vous permet de vous connecter à SSH sans saisir de mot de passe à chaque fois. Utilisez les commandes suivantes sur votre poste de travail Admin :

  • ssh-keygen Acceptez le dossier par défaut et utilisez une phrase de passe vide. Faites cette étape une fois .
  • ssh-copy-id -i ~/.ssh/id_rsa.pub gef330@<Host IP Address>Entrez le mot de passe par défaut du cours lorsque vous y êtes invité. Effectuez cette étape pour chaque hôte auquel vous accédez avec SSH.
  • Connectez-vous à la boîte cible avec ssh gef330@<Host IP Address>.
    • Si vous êtes toujours invité à entrer un mot de passe lors de la connexion SSH de votre poste de travail d'administration vers n'importe quel autre boîtier de votre réseau, cela n'a pas fonctionné. Corrigez-le avant de faire tester votre réseau.
  • Si l' instructeur est invité à entrer des mots de passe lors de la tentative de test de votre réseau, vous serez pénalisés en raison d'une perte de temps. Prenez le nombre de groupes et multipliez-le par le nombre de machines virtuelles par groupe, puis déterminez si vous souhaitez saisir des mots de passe autant de fois.

Remarque: il est essentiel que vos postes de travail administratifs soient correctement configurés. Si l'instructeur est incapable de SSH sur les postes de travail administratifs ou si les postes de travail administratifs ne peuvent pas SSH sur les autres postes de travail et serveurs sur le port TCP 22, le script de test échouera et vous ne recevrez pas de notes pour cette partie du réseau. (L'instructeur sera SSH à partir de 10.12.0.166 ou du serveur commun.)

Persistance

Toutes les modifications apportées aux serveurs sont effectuées à l'aide des commandes du terminal ( sudo ip ____) et sont non persistantes, c'est-à-dire qu'elles ne survivront pas à un redémarrage du système. Il existe plusieurs façons d'obtenir la persistance. Il existe des fichiers de configuration réseau distincts qui peuvent être modifiés pour le routage persistant et la configuration de l'interface, ou vous pouvez faire exécuter automatiquement certaines commandes au démarrage ou à la connexion.

Dans ce cas, la meilleure option est de configurer votre VM pour exécuter vos scripts au démarrage. Chaque utilisateur Linux a en fait un fichier caché appelé .profilequi est exécuté à chaque fois que l'utilisateur se connecte. Normalement, cela ne nous aiderait pas à exécuter quelque chose au démarrage, mais comme vous l'avez peut-être remarqué, vos machines virtuelles Linux se connectent automatiquement au compte  gef330. Ainsi, tous les scripts que nous ajoutons ici seront exécutés au démarrage de votre VM.

Note: Le . au début de  .profile le marque comme un fichier caché. Si vous le recherchez dans un explorateur de fichiers GUI, vous devrez activer l'affichage des fichiers cachés. Si vous le recherchez à partir de la ligne de commande, ls -lalistera tous les fichiers, y compris les fichiers cachés.

  • Utilisez un éditeur de texte pour ouvrir /home/gef330/.profile
  • Faites défiler vers le bas du fichier et ajoutez le chemin d'accès complet au fichier de script.
    • Si votre script est appelé setup.shet est stocké sur votre bureau, ajoutez-le /home/gef330/Desktop/setup.shau fichier. Sur Linux, le chemin est sensible à la casse!
  • Enregistrez le fichier et redémarrez votre VM. Au démarrage de la sauvegarde, confirmez que votre script a été exécuté et que vos configurations réseau ont été appliquées.
