Labo 2: Introduction à la configuration réseau et à la capture de paquets

GEF330 hiver 2021

Objectifs

Ce laboratoire vise à:

  • Vous donner l’information nécessaire à la configuration de réseau typique pour un hôte
  • Vous introduire à la capture de paquet
  • Vous donnez les outils nécessaire à faire l’analyse de paquets rudimentaire

Introduction

Ce laboratoire est une introduction supplémentaire à l'environnement de laboratoire au sein de l'environnement virtuel du laboratoire de sécurité des réseaux informatiques. Vous allez configurer un petit réseau composé de quelques ordinateurs ordinaires (mais virtuels!), d'un ordinateur central hébergeant un serveur telnet, un serveur ftp et un serveur Web, un concentrateur (hub) Ethernet et un standard de connexion (patch panel). Vous utiliserez aussi un outil de reniflage qui vous permettra de faire des captures de paquets du trafic étant diffusé sur le réseau. L’analyse de ces captures de trafic vous aidera à comprendre la structure des communications réseaux basées sur les datagrammes et de l’encapsulation de protocoles.

Bâtir un petit réseau

Figure 1 - Configuration réseau

  • Pour ce labo, vous aurez besoin d’une connexion entre votre réseau d’équipe (Team Network) et le réseau commun (Common Network) comme indiqué à la figure 1. Dans le vrai laboratoire, vous auriez dû utiliser un standard de connexion réseau et un concentrateur pour connecter votre commutateur virtuel externe (External Virtual Switch) au concentrateur commun (Common Hub), mais dans l'environnement virtuel, cette étape est déjà effectuée pour vous.
    • Notez que chaque paire (ou trio) d'étudiants du laboratoire se connecte au même concentrateur (hub) commun.
    • Rappelez-vous que x est votre numéro d'utilisateur.
    • Notez que bien qu'il soit étiqueté comme un commutateur, la vraie histoire derrière le commutateur virtuel externe est un peu plus compliquée. Dans ce cas, il a été configuré pour agir davantage comme un concentrateur afin de fournir les fonctionnalités requises dans ce laboratoire. Une discussion plus approfondie nécessite de rentrer dans des détails VMware qu'il vaut mieux éviter à ce stade.
  • Démarrez la Windows VM -Le réseau commun est un sous-réseau décrit par l’adresse IP 10.12.0.0/16. C’est à dire que les premiers 16 bits des adresses IP sur ce réseau décrivent l’adresse de réseau, et que les 16 dernier bits décrivent l’adresse des ordinateurs. Le masque de sous-réseau indique à l’ordinateur quels bits correspondent à l’adresse réseau. Ceci signifie que le masque de sous-réseau pour ce réseau est 255.255.0.0, ce qui donne en binaire 11111111.11111111.0.0. 16 bits.
    • Vous devez maintenant configurer les adaptateurs de la Windows VM pour communiquer sur ce réseaux en fixant les adresses IP et masques de sous-réseau (netmask) selon les instructions suivantes.
      • Bien que vous n’inter-agisserez pas avec d’autres équipes lors de ce labo, vous devez quand même être certain de ne pas utiliser la même adresse IP qu’un autre étudiant puisque chaque ordinateur doit avoir une adresse unique sur un réseau.
      • Pour ce faire, vous allez utiliser 10.12.x.20 comme adresse IP unique de votre Windows VM.
      • Qu’est-ce que vous penser devoir utiliser comme masque de sous-réseau? Notez la réponse sur votre rapport.
  • Configurez les adaptateurs réseau
    • pour chaque labo, si vous oubliez comment accéder à certaines applications que nous utilisons fréquemment, faite référence au préambule. Par exemple, on configure un adaptateur ainsi:
      • Start -> Control Panel -> Network and Sharing Center-> Change Adapter Settings (left hand menu)
    • Assurez-vous que l’adaptateur Internal est éteint.
      • s’il ne l’est pas, vous pouvez le faire en faisant :
      • Clic Droit -> Disable
    • Allumer et configurer l’adaptateur External (Pourrait s’appeler : Local Area Connection)
      • Clic-Droit -> Enable (si besoin)
      • Clic-droit -> Properties
    • Sous l’onglet General, sélectionnez Properties -> Internet Protocol (TCP/IP) -> Properties et choisissez “Use the following IP address:”
      • Entrez votre adresse IP et votre masque de sous-réseau.
      • Ne vous souciez pas de la passerelle (Gateway) ou du serveur de noms (DNS) pour ce labo.
  • Testez votre connexion réseau
    • Ouvrez une invite de commande et vérifiez votre configuration réseau :
      • ipconfig /all
    • Confirmez votre connexion au réseau en faisant ping à votre ordinateur, au Server VM et à l’imprimante (Printer)
      • ping 10.12.x.20 - pour être certain que vous pouvez communiquer avec votre propre ordinateur
      • ping 10.12.x.1 - pour être certain que vous pouvez communiquer avec le serveur
      • ping 10.12.0.9 - pour être certain que vous pouvez communiquer avec le serveur SFTP. (Non montré dans la figure 1)
    • Un démon telnet fonctionne sur le Server VM . Utilisez telnet pour vous inscrire au compte d’alice sur le Server VM
    • Ouvrez une session à telnet avec account : alice et password : secret. Utilisez telnet -h ou bien man telnet si vous avez besoin de détails.
    • Fermez votre session avec: logout
  • Votre petit réseau est maintenant configuré pour bien fonctionner.

Analyse de captures de paquets

L’outil windump est une conversion de l’outil tcpdump qui est très populaire sur les systèmes Linux. Vous allez utilisez windump pour capturer et analyser du trafic réseau. Démarrez windump d’une invite de commande.

  • Exécutez tout d’abord windump -D pour découvrir les noms et numéros des interfaces réseaux de votre Windows VM
    • Notez qu’il faut respecter les majuscules et minuscules.
    • Vous remarquerez que les adaptateurs listés par Windows sont difficiles à interpréter.
    • Confirmer qu’il y a seulement un adaptateur listé; si vous avez bien suivi les étapes précédentes, celui-ci devrait être l’adaptateur External. Notez les quatre premiers octet du device name dans votre rapport de labo.
    • Allumez l’adaptateur Internal et exécutez windump -D de nouveau; vous devriez maintenant être en mesure de noter les quatre premier octets de celui-ci dans votre rapport
    • Répétez cette procédure à chaque fois que vous désirez identifier les adaptateurs listés par Windows sous windump. La bonne nouvelle est que l’équivalent Linux de windump, appelé tcpdump identifie les interfaces bien plus clairement.
  • Nous utilisons l’option -i pour spécifier l’adaptateur à utiliser comme suit: windump -i 2 (si nous voulions utiliser l’adaptateur 2)
  • Nous utilisons seulement l’adaptateur External aujoud’hui, vous pouvez donc éteindre l’adaptateur Internal. Puisque nous utilisons seulement un adaptateur aujourd’hui, nous n’avons pas à utiliser l’option -i; windump utilisera l’adaptateur External par défaut.
  • Invoquez maintenant windump -n. L'option -n sert à supprimer les consultation de tables pour les numéros de port et d’ordinateur.
  • (#1) Soyez bien certain de comprendre pourquoi on utilise -n. Qu’est-ce que windump fait pour chaque paquet lu si on n’utilise pas l’option -n ? (Rappelez vous que la discussion est l’aspect le plus important du rapport).

    telnet 10.12.x.1

Ce trafic est probablement associé avec les services du système d’exploitation Windows. Vous verrez peut être des paquets associé avec l’activité d’autres étudiants.
  • (#2)Enregistrez quelques adresses IP des ordinateurs d’autres étudiants trouvés dans votre capture de trafic et expliquez avec qui ceux-ci communiquaient.
  • Vous remarquerez que le trafic présenté par windump est très difficile à lire; on aurait avantage à le filtrer. On peut utiliser plusieurs expressions avec windump de la ligne de commande lors de l’invocation du programme. Les expressions permettent à l’utilisateur de filtrer le trafic et de ne rapporter que ce qui est utile pour une expérience donnée. Par exemple, on peut filtrer tout le trafic associé avec votre session telnet au Server VM en sachant que telnet utilise le port reconnu 23 au serveur. On peut donc filtrer pour exclure tout trafic sauf notre session telnet en spécifiant:

    • host "10.12.x.20 and port 23"

  • lorsqu’on invoque le programme de la ligne de commande. Notez que l’ordre de la commande doit suivre la convention suivante :

    • command <switch> <option> <filter>

  • Essayez ceci et observez les résultats; n’oubliez pas d’inclure les options -i et -n. Vous devriez maintenant voir seulement votre trafic telnet.
  • Le programme windump a des options permettant de faire rapport d’autre information réseau intéressante lorsqu’il rapporte les résultats à l’écran. Pour voir une liste complète de ces options, jetez un coup d’oûtil à la page manuel de windump.

    • Si vous voulez avoir accès à une version du manuel dans le CNSl, démarrez votre Linux VM. Ouvrez un terminal et utilisez la commande man pour avoir accès au manuel pour différents utilitaires. Dans ce cas, pour voir le manuel de tcpdump (l’équivalent linux de windump), tapez :

    man tcpdump

  • (#4) Lorsqu’on utilise l’option -s, comment peut-on s’assurer de capturer les paquets en entier? Experimentez avec snaplen afin d’être bien certain de comprendre son utilisation.
  • Invoquez windump de nouveau pour filtre votre trafic telnet. Utilisez les bonnes options pour capturer le paquet en entier, fournir la forme hexadécimal brute ainsi que la représentation ASCII de l’hexadécimal. Cette capture vous donne une vue de chaque octet transmis dans un paquet IP. La traduction ASCII peut parfois être utile puisque la charge utile d’une session telnet contient de l’information ASCII. Par exemple, si vous listez le contenu d’un répertoire sur votre session telnet, vous pourrez facilement lire le contenu sous windump. Par contre, les données associées avec les en-têtes TCP et IP sont encodées numériquement et n’ont aucune signification ASCII. windump essaie quand même de faire un décodage ASCII. Ces formats ASCII et hex peuvent être utile pour comprendre les données et pour déboguer les protocoles réseaux.
